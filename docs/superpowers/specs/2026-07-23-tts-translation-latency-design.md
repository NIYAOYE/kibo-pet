# 多语种 TTS 翻译链路低延迟方案

> 日期：2026-07-23  
> 状态：方案探索，尚未进入实现  
> 推荐方案：保留原文 UI 的前提下，采用“流式翻译 → 稳定短语 → 流式 TTS”的流水线。

## 1. 背景与问题

当前多语种朗读的流程为：LLM 产出原文 → 按段切分 → 用 LLM 翻译成目标语言 → 等待完整译文 → 将译文提交给本地 TTS sidecar → 等待首个 PCM 音频块 → 播放并显示原文。

代码中的翻译器虽然通过 `streamChat()` 接收流式数据，但会把所有文本累积为一个字符串后才返回；因此它对下游仍是一次完整、阻塞式的翻译调用。`voiceProvider` 也会等待翻译成功后才调用 `sidecar.speak()`。当聊天文字与首段音频同步显示时，用户感知到的首句延迟会叠加为：

```text
原文达到可切分边界的等待
  + 翻译模型输出整段译文的等待
  + TTS 首包推理时间
  + 播放缓冲时间
```

如果保持默认的“batch + smart”播放设置，语音任务还会等到完整回复结束或等到智能分段凑够长度，进一步放大延迟。

相关现状：

- `src/main/voice/translate.ts`：翻译流被完整收集后才返回。
- `src/main/voice/voiceProvider.ts`：翻译完成后才发起 TTS 请求。
- `src/main/shell/replyPresenter.ts`：支持 `stream`/`batch` 播放触发与 `sentence`/`smart` 文本分段。
- `src/main/voice/speechSequencer.ts`：已具备两项预取、严格按原文顺序转发 PCM 的能力。
- `resources/voice/gsv_server.py` 和 `resources/voice/genie_server.py`：均能用 SSE 持续输出 PCM；GSV 还能选择 token 或 sentence 推理流。

## 2. 目标与约束

### 目标

1. 在“屏幕显示原语言、语音朗读目标语言”的产品语义下，尽可能缩短从 LLM 首个可用文本到首个可播放音频的时间。
2. 保留现有的语音顺序保证、取消、失败回退、音频与文字同步呈现能力。
3. 复用当前 GSV-TTS-Lite / Genie-TTS sidecar 与 PCM/SSE 协议，不以更换 TTS 后端作为前置条件。
4. 让用户能通过现有设置先获得无代码的感知延迟改善。

### 非目标

- 不在本方案中改变角色音色、宠物包格式、音频 IPC 格式或口型同步协议。
- 不承诺逐 token 翻译即刻朗读；翻译语言可能发生语序重排，必须先确认目标文本边界稳定。
- 不将端到端语音翻译/语音克隆模型作为首期依赖。

## 3. 可选路线

| 路线 | 说明 | 首句延迟 | 主要代价 | 结论 |
| --- | --- | --- | --- | --- |
| A. 直接以目标语言回答 | 在主聊天请求中要求 LLM 直接生成日语/英语/中文，TTS 直接朗读，无翻译调用。 | 最低 | UI 原文会变成目标语言；若仍需原文，需要额外生成或显示策略。 | 若产品允许 UI 与语音统一为目标语言，这是最简单且最快的路线。 |
| B. 流式翻译 + 流式 TTS | 保留原文 UI。译文流按稳定短语释放给 TTS，翻译后半段与前半段合成并行。 | 低 | 需要处理目标语稳定边界、并发、取消和顺序。 | **推荐。** 在不改变产品语义下收益最大。 |
| C. 专用机器翻译服务/本地 NMT | 将翻译从通用聊天 LLM 换成专门的翻译 API 或本地小模型；按句/短语调用 TTS。 | 中低、可预测 | 引入网络费用或额外模型、显存和运维。 | 可作为 B 的翻译器实现选项，而不是独立架构。 |
| D. 端到端“翻译并合成”模型 | 一个模型直接将源语言文本变为目标语言语音。 | 理论上低 | 角色音色一致性、跨语言质量、可打包性和调优难度都较高。 | 不建议作为首期方案。 |

## 4. 推荐架构：流式翻译 → 稳定短语 → 流式 TTS

### 4.1 数据流

```text
LLM 原文 token 流
        │
        ▼
原文句子切分器（仅在可确认句末时提交）
        │  source segment
        ▼
流式翻译器
        │  target token 流
        ▼
目标语言稳定短语切分器
        │  target phrase
        ▼
TTS sidecar（SSE PCM 流）
        │
        ▼
现有 SpeechSequencer（原文段顺序闸门）
        │
        ├── 放行对应原文 UI
        └── 转发 PCM 给播放器
```

核心原则是：翻译输出不再等到整段完成才交给 TTS；但也不直接朗读任意翻译 token。只有到达目标语言中可安全截断的边界时，才将该短语作为一个 TTS 任务提交。

### 4.2 稳定边界规则

第一期采用保守策略，避免把后续可能改写的翻译半句提前读出：

1. 原文侧仍由现有句子切分器产生输入段，不对尚未闭合的原文句子进行翻译。
2. 译文侧优先在 `。！？.!?` 等句末处切分；对中文和日文可额外识别 `、`/`,` 后的足够长短语。
3. 设一个较小的最短朗读长度，例如 8–16 个可朗读字符；过短短语继续缓冲，避免频繁起音和断裂语调。
4. 翻译请求结束时，强制 flush 剩余译文。若没有任何稳定边界，也必须确保整段可被朗读或按失败策略回退。
5. 翻译器必须禁止解释、标题、Markdown、原文复述，只返回目标译文；否则稳定边界会产生错误音频内容。

这会牺牲少量“最早一个 token”的理论速度，换来可理解的语义和更自然的停顿。实际首句通常只需等翻译器产出第一个短语，而不再等待完整译文。

### 4.3 顺序与展示语义

现有 `speechSequencer` 已经以源文本段序号维护播放顺序。推荐继续让它拥有唯一的“放行权”：

- 每个原文段对应一个逻辑语音项，而一个逻辑项可以包含多个目标语言 TTS 短语。
- 原文 UI 只在该逻辑项的第一个 PCM 真正按顺序转发时显示一次。
- 后续短语可连续发送 PCM，不能跨越前一个原文段的未完成音频。
- 后续段可以预取翻译或预合成，但音频必须继续在现有顺序闸门中缓冲。
- 翻译失败、目标文本为空或 TTS 失败时，立刻按现有 display-only 行为放行原文，绝不阻塞后续段。

这样既可保留“文本显示与第一声同步”的体验，也避免多句并发时出现语音和气泡乱序。

### 4.4 取消与背压

一个回复轮次需有独立的 `AbortSignal`/generation：

- 用户取消、开始新回复、切换宠物或销毁会话时，同时中止在途翻译、未开始的 TTS 短语及 sidecar 请求。
- 任何旧 generation 的翻译 token、PCM 或显示回调都必须被忽略。
- 为避免翻译过快而堆积音频，维持有限预取窗口：当前原文段与下一原文段最多并行；每段的目标短语也需设队列上限。
- 当队列达到上限时暂停读取/处理后续翻译流，或只保留尚未合成的文本缓冲，避免内存增长与过时语音。

## 5. 分阶段落地建议

### 阶段 0：无代码配置验证

先在真实机器上将以下设置作为低延迟预设进行对照测试：

- 播放触发：`stream`
- 文本分段：`sentence`
- GSV 合成分块：`token`
- 设备：优先 CUDA；如硬件和运行时支持，再测试 Flash Attention。

这不会消除翻译等待，但能消除“等整条回复结束才入队”的等待，并使 GSV 尽早产出首个 PCM。Genie 后端是否同样改善，需要实测，因为它的 sidecar 内部流式策略与 GSV 不同。

### 阶段 1：可观测性与基线

在不改变行为的前提下，为每个段记录单调时间戳：

| 指标 | 含义 |
| --- | --- |
| `t_llm_first_text` | 本轮 LLM 首个文本 token 到达 |
| `t_source_segment_ready` | 原文段达到切分边界 |
| `t_translation_first_token` | 翻译器首个 token 到达 |
| `t_translation_first_stable_phrase` | 首个可提交 TTS 的目标短语到达 |
| `t_tts_request` | 首次提交 TTS 请求 |
| `t_first_pcm` | 首个 PCM 到达主进程 |
| `t_pcm_forwarded` | 首个 PCM 实际转发播放器 |

计算并观察：

```text
翻译首包 = t_translation_first_token - t_source_segment_ready
可朗读翻译等待 = t_translation_first_stable_phrase - t_source_segment_ready
TTS 首包 = t_first_pcm - t_tts_request
端到端首声 = t_pcm_forwarded - t_llm_first_text
```

只有先得到 GSV/Genie、CUDA/CPU、不同目标语言下的 p50/p95 基线，才能判断主要瓶颈是否真是翻译，而不是模型冷启动、GPU 回退或分段策略。

### 阶段 2：流式翻译器契约

将当前“`translate(): Promise<string>`”演进为可取消的异步翻译流，例如概念上的：

```ts
interface StreamingTranslator {
  translateStream(
    text: string,
    target: 'zh' | 'ja' | 'en',
    signal: AbortSignal
  ): AsyncIterable<string>
}
```

实现要求：

- 现有基于 `streamChat()` 的实现直接逐块产出目标文本，而不再先累积为完整字符串。
- 保留一个收集型适配器，用于不支持翻译流的提供方或故障回退。
- 将网络错误、空结果、取消、上游 `error` chunk 区分为明确结果，不能使队列悬挂。

### 阶段 3：目标语言短语切分与单段多短语合成

新增纯函数的目标语言短语缓冲器；它消费翻译 token 流并产出稳定短语。将单个原文段映射为“零到多个 TTS 短语”的逻辑任务，并保持原文段级的显示与顺序语义。

建议先只实现“句末 + 最短长度”规则，收集真实样本后再考虑逗号级切分。不要一开始做 token 级猜测或激进的词典规则。

### 阶段 4：翻译器选择与降级

保留一个统一翻译接口，允许后续接入：

1. 当前聊天 LLM：零新增运行时，质量与上下文最好，但首 token 可能受远程推理波动影响。
2. 专用云翻译：延迟和术语质量较稳定，但需管理密钥、费用和网络失败。
3. 本地小型 NMT：离线可用、隐私更好，但要评估 CPU/GPU 抢占、模型体积和冷启动。

第一期不必在 UI 暴露多供应商选择；只需把边界设计成可替换，待基线数据证明收益后再决定是否引入新后端。

## 6. 验收标准

### 功能正确性

- 原文 UI 与聊天记录保持原样；目标语言译文只进入 TTS，不回写用户可见原文。
- 多段回复的原文显示、音频播放顺序始终与原始 LLM 输出一致。
- 翻译流、TTS 流、取消和宠物切换的旧回调均不会泄漏到新回复。
- 翻译或 TTS 任一失败时，该段原文可见，后续段仍会继续处理。
- `targetLanguage: 'auto'` 或无需翻译的文本沿用现有快速路径，不额外经过翻译流水线。

### 性能

- 在相同硬件、相同 LLM、相同宠物与相同回复样本下，流式模式的端到端首声 p50 应明显低于当前完整翻译后合成基线；建议以至少 30% 的降低作为进入默认设置讨论的门槛。
- p95 不应因预取而出现无界排队；长回复的内存占用和未播放音频缓冲应有明确上限。
- GSV 与 Genie 均需分别验证，CPU 与 CUDA 至少各覆盖一次。

## 7. 风险与应对

| 风险 | 应对 |
| --- | --- |
| 翻译后文改写前文，提前朗读造成语义错误 | 只在保守的目标语言稳定边界提交 TTS；先句末，后续再以真实数据评估短语级切分。 |
| 短语过短导致断句生硬、首包增多 | 设置最短长度和合并窗口；必要时按语言配置阈值。 |
| 翻译与 TTS 同时占用同一远程/本地模型资源 | 有界预取、按段取消；采集 p95 与 GPU 利用率后调整并发窗口。 |
| UI 被首音频门控，翻译慢时文字也迟到 | 保持当前“音频不可用即显示原文”的回退；产品若更重视阅读速度，可将 UI 门控改为独立策略，但那是另一项体验取舍。 |
| 更换翻译后端增加隐私、费用或安装复杂度 | 先将接口抽象为可替换，默认仍使用当前提供方；新后端以显式选择方式引入。 |

## 8. 决策建议

默认选择路线 B：先把当前 LLM 翻译从“收完整译文再合成”改为可取消的翻译流，再以稳定短语喂给既有 SSE TTS。该路线复用当前最成熟的组件，保持原文 UI、角色音色、顺序和错误回退，也能覆盖 GSV 与 Genie 两个后端。

在投入实现前，应先确认产品决策：**是否必须保留原语言的气泡/聊天文本？**

- 若“不必保留”，路线 A 可直接让主 LLM 以目标语言回答，成本最低、首句最快。
- 若“必须保留”，采用本文件推荐的路线 B，并先完成阶段 0–1 的真实基线测量。
