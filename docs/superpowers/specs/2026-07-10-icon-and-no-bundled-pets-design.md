# App icon replacement + stop bundling pet packages

Two small, independent changes requested together:

1. Replace `build/icon.ico` with an original, procedurally-drawn icon (inspired by, but not derived from, a reference image the user can't redistribute for copyright reasons).
2. Stop shipping any real pet package inside `pnpm dist` output; when a fresh install has zero pets available, guide the user to import one instead of crashing.

## Part 1 — App icon

### Problem

`build/icon.ico` is currently generated by `tools/hatch-desktop-pet/scripts/make_app_icon.py` from the `luluka` pet's idle sprite frame. The user wants the app icon to instead evoke a reference image they shared (a cute robot-cat-astronaut mascot) — but that reference image itself cannot be reused/redistributed (copyright/provenance unclear), and this session has no image-generation tool available. The icon must therefore be an **original, code-drawn** asset that is stylistically similar in spirit (friendly robot-cat head, glowing features, rounded badge) but not a reproduction of the reference's pixels.

### Design

New one-off Python script, `tools/hatch-desktop-pet/scripts/make_app_icon_original.py` (same `conda run -n peticon python ...` convention as the existing script; does not touch or depend on any pet package):

- Canvas: 256×256, built entirely from primitive shapes via PIL (`ImageDraw`), then downsampled to the standard icon sizes.
- Background: rounded-square badge, purple→blue radial/vertical gradient (matches the reference's color mood, not its exact gradient).
- Head: white rounded "helmet" silhouette with two triangular cat ears (outline only, flat-shaded — no attempt at the reference's shading/highlights).
- Antennae: two thin stalks topped with small glowing cyan circles.
- Face: a dark visor shape with two glowing cyan closed/curved "happy" eyes (simple arcs), no mouth/whiskers/other reference-specific detail.
- Output: `build/icon.ico` with sizes `[(16,16),(32,32),(48,48),(64,64),(128,128),(256,256)]`, same as today.

The existing `make_app_icon.py` (luluka-derived) is left in place but is no longer the source of `build/icon.ico` — it's now dead/legacy relative to the shipped icon since it depends on a pet package that may not exist on a given machine post Part 2. It is not deleted (out of scope; not asked for), just superseded.

### Testing

Visual/asset generation script — no unit tests apply (matches the precedent of the existing `make_app_icon.py`, which also has none). Verification is running the script and visually inspecting `build/icon.ico` (e.g. opening it, or checking it renders correctly in a packaged build's taskbar/installer per `electron-builder.yml`'s `win.icon: build/icon.ico`).

## Part 2 — No default-bundled pet packages + first-run import prompt

### Problem

`electron-builder.yml`'s `extraResources` currently copies the entire `pets/` directory (all pet packages present on the build machine's disk — `luluka`, `youka`, `alice`, etc.) into every packaged installer. The user wants `pnpm dist` builds to ship with **zero** pet packages by default; anyone installing the app must import one themselves via the existing "导入宠物包" flow (built in MVP-09).

This breaks the current startup assumption: `startShell()` resolves `activePetId` (settings, default `'luluka'`) to a pet home via `ensurePetHome()`, falling back to the default id if the configured one is missing — but today, if the **default** id is also missing, it re-throws, and `src/main/index.ts`'s top-level catch shows a raw error-stack `dialog.showErrorBox`. Once nothing is bundled, this is what *every* fresh install would hit on first launch.

### Design

**2a. Packaging — `electron-builder.yml`**

Change the `pets` entry under `extraResources` so only `pets/.keep` is copied, not any pet subfolder:

```yaml
extraResources:
  - from: pets
    to: pets
    filter:
      - '!**/*'
      - '.keep'
  - from: skills
    to: skills
  - from: resources
    to: resources
```

Update the adjacent comment (currently says "pets: 全部磁盘宠物(luluka/youka...)…首启播种到 userData") to reflect that packaged builds ship no pets and rely on user import.

This only affects `pnpm dist` (packaged) output. `pnpm dev` / `pnpm preview` resolve `appRoot` to the repo root (`src/main/shell/index.ts`: `app.isPackaged ? process.resourcesPath : join(dirname, '../..')`), so local development continues to see every pet folder present on disk, unaffected.

**2b. Startup — graceful "zero pets" onboarding path**

In `src/main/shell/index.ts`, keep the existing two-step resolution (configured `activePetId` → fall back to default `'luluka'`) exactly as-is. Only when **both** fail — meaning no bundled pet and nothing yet imported into `userData/pets/` — branch into a new onboarding path instead of re-throwing:

- Skip creating the pet sprite window, dialog window, bubble window, todo window, idle watcher, memory manager, agent providers/chat store, automation/browser-control controllers, voice sidecar — none of these can meaningfully exist without a resolved pet home.
- Create the system tray (reusing `createTray`, so the app is never an unreachable background process — same "退出" affordance as normal boot; the other two tray actions can simply open Settings too, since nothing else exists yet).
- Create and immediately show the existing Settings window (`createSettingsWindow`) — no new window/UI is built.
- Register a **minimal, self-contained** set of IPC handlers sufficient for the Settings page to load and be fully usable in this mode:
  - `GET_SETTINGS` / `SET_SETTINGS` — same shape as the normal handlers, backed by the same `loadSettings`/`saveSettings`/`normalizeSettings` and the same 4 secret stores (`secrets`, `searchSecrets`, `embeddingSecrets`, `firecrawlSecrets`, constructed the same way). `SET_SETTINGS` is a simplified copy that omits the `browserControl.close()` side effect the normal handler has (no live `browserControl` controller exists yet in this mode — irrelevant anyway since browser control can't have been running).
  - `SET_API_KEY`, `SET_SEARCH_KEY`, `SET_EMBEDDING_KEY`, `SET_FIRECRAWL_KEY` — identical to the normal handlers (only depend on the secret stores).
  - `TEST_CONNECTION` — identical to the normal handler (pure network call, no pet dependency).
  - `LIST_PETS`, `IMPORT_PET` — identical to the normal handlers (only depend on `bundledPetsDir`/`userPetsDir`).
  - `RELAUNCH_APP` — identical to the normal handler.
  - `CONFIRM_DESKTOP_CONTROL`, `CONFIRM_BROWSER_CONTROL`, `CONFIRM_CDP_MODE`, `OPEN_MEMORY_DIR` are **not** registered in this mode (genuinely pet/feature-dependent: confirm dialogs gate features that can't run yet; memory dir is per-pet). If the user toggles the corresponding checkboxes on the Settings page before importing a pet, the confirm-dialog IPC call rejects silently (unhandled promise rejection, logged to console) and the checkbox simply doesn't get gated — a cosmetic inconsistency, not a crash, and not reachable in the intended flow (user is being directed to import a pet first). Once a pet is imported and the user relaunches, the app boots normally with the full handler set and these work as today.
- `GET_SETTINGS`'s response gains one new field so the renderer can show a banner: compute `noPetInstalled: listPets(petCatalogDirs).length === 0` (reuse the same `listPets` used for `LIST_PETS`) and add it to `SettingsSnapshot` in `src/shared/ipc.ts`. In the normal (non-onboarding) boot path, this is simply always `false` (a pet was just resolved, so at least one exists) — no extra computation needed there.
- `settings.ts`/`settings.html`: when `snap.noPetInstalled` is true, show a visible banner near the pet picker/import section (e.g. "未检测到宠物包,请先导入一个宠物包后重启"), styled to draw the eye (reuse existing `status`-line conventions rather than inventing new CSS where possible). The rest of the page (Provider/API Key/搜索后端/etc.) remains visible and fully functional per the user's explicit choice — no tab hiding.
- After a successful import, the existing "切换宠物需要重启" flow (`relaunchBtn`, already built in `settings.ts`) applies unchanged: user selects the imported pet, saves, sees the relaunch button, clicks it, app restarts and now resolves that pet home normally through the untouched 2-step logic in 2b's first paragraph.

### Non-goals / out of scope

- No zip-based pet import (folder-picker import already exists and is reused as-is).
- No changes to `ensurePetHome`/`petCatalog.ts` logic — reused unchanged.
- No auto-import of anything on first run — the user must explicitly pick a folder via the existing dialog.
- Not hiding/restructuring unrelated Settings sections in onboarding mode (confirmed with user: keep everything visible/usable).
- Not deleting the luluka-derived `make_app_icon.py` script.

### Testing

- `src/main/shell/index.ts`'s onboarding branch: covered by a focused unit test if the branching logic can be isolated into a small testable function (e.g. "given configured+default pet resolution both fail, and existing settings snapshot behavior, does `GET_SETTINGS` report `noPetInstalled: true`"); the window/tray creation itself is Electron GUI wiring, verified by running the packaged (or `pnpm preview` with an emptied local `pets/` dir) app, per this repo's existing test-only-pure-logic convention.
- `electron-builder.yml` change: verified by running `pnpm dist` and inspecting the output installer/`resources/pets` folder contains only `.keep` (or via `dist/win-unpacked/resources/pets` after an unpacked build) — not unit-testable (build config).
- `src/shared/ipc.ts` `SettingsSnapshot.noPetInstalled` field: existing IPC-shape tests (if any) updated to include the field.
