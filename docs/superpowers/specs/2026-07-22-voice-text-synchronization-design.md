# 语音与文字逐段同步设计

## 目标

让宠物的聊天回复以“已可播放的语音片段”为文字显示闸门：当某个切分段的第一个 PCM 音频块将按播放顺序转发给渲染进程时，才显示该段对应的原始 LLM 文本。

LLM 原文始终完整显示；仅供 TTS 使用的派生文本会在本地清洗。语音不可用或某段无法产生音频时，文字不被隐藏，按原顺序直接放行。

## 范围与非目标

本次覆盖常规聊天与快捷文字加工动作的 LLM 回复。不会改变播放器、口型驱动、模型安装、宠物包格式或 IPC 音频格式。

“同步”指文字在第一个可播放 PCM 块被发往播放器的同一顺序点显示；不等待声音实际播放结束，也不等待整段 TTS 请求完成。这样文字出现与声音开始相邻，同时保留现有低延迟播放。

## 行为规则

### 语音可用性

每次回复开始时检查语音门面是否真正可用：TTS 已开启、当前宠物具有所选后端的语音模型、对应运行时已安装，且 sidecar 已经启动并创建了语音队列。若任一条件不满足，聊天保持现有行为：LLM 文本块到达即通过 `pushStream` 显示，且不创建任何语音任务。

回复进行中若某一段翻译或 TTS 推理失败、未返回音频，或清洗后没有可朗读字符，则该段按“无可用音频”处理。它不会阻塞后续文字，原始文本会在正确的相对顺序中放行。

### 文本、切分与显示

LLM 原始文本先累积为完整回复，并由当前 `textSplit` 策略划分为语音段。`playbackTrigger: 'stream'` 在段完整时立即入队；`playbackTrigger: 'batch'` 在 LLM 回复结束后用同一策略划分并入队。因此两个设置都遵守逐段同步，只区别语音任务何时开始。

每个待朗读项保存两份值：

- `displayText`：未经改写的 LLM 原始段，用于聊天气泡与记忆记录。
- `speakText`：仅供 TTS 使用的清洗结果，绝不回写到气泡、聊天记录或记忆。

语音队列保持现有“最多两项预合成、严格按原始顺序转发音频”的能力。某项第一个 PCM 块真正轮到转发时，队列先调用其一次性 `onDisplay` 回调，再转发该 PCM 块。即使第二项先完成推理，它的音频和文字都会缓冲至前一项已经放行，避免播放与文字乱序。

回复收尾不会直接用完整消息刷新对话。聊天层会等待本轮所有待显示项都已放行，随后才追加完整原文到 memory、推送最终消息列表、发送 `pushDone`。如果是带部分文本的错误，先等待已生成段放行，再保存部分原文并报告错误。取消或开始新回复会使旧轮次失效，旧音频与延迟文字不得泄漏到新回复中。

### TTS 预清洗器

预清洗器是纯本地、确定性的函数；不额外调用 LLM、网络服务或模型。它在现有 Markdown 可朗读化逻辑基础上生成 `speakText`，规则偏保守：保留自然语言、数字和单位，只移除明确不值得朗读的技术载荷。

它将：

- 保留标题、列表文字、普通段落、Markdown 链接的可见标签和表格的数据单元格。
- 移除裸 `http://`、`https://`、`www.` URL，Markdown 链接目标、图片链接、`mailto:` 和数据 URL。
- 移除围栏代码、行内代码、HTML 注释及标签、表格分隔线和 Markdown 排版符号。
- 移除独立的命令提示符、纯文件路径、长哈希、UUID 与疑似密钥等纯技术标识；清理后折叠多余空白。
- 在“自然语言加 URL/代码”的混排段中只朗读自然语言；在“仅 URL/代码/技术标识”的段中得到空结果，不发起 TTS，直接显示原始段。

例如 `文档见 https://example.com/a` 的气泡仍完整显示，而 TTS 只接收“文档见”。仅包含 `https://example.com/a` 的段不调用 TTS，却会照常显示。

## 组件边界

| 组件 | 责任 |
| --- | --- |
| `src/main/voice/speakableText.ts` | 将原始段确定性清洗为朗读文本，并识别清洗后为空的段。 |
| `src/main/voice/voiceProvider.ts` | 将单项处理结果明确返回为“产生音频”“跳过”或“失败”，不让失败静默阻塞队列。 |
| `src/main/voice/speechSequencer.ts` | 维持预合成及顺序缓冲；在首个可播放 PCM 转发前放行对应显示回调，并对跳过/失败项按顺序放行。 |
| `src/main/shell/petSession.ts` | 暴露能准确反映语音队列是否已就绪的门面，避免 sidecar 尚未启动时把聊天误判为可语音。 |
| `src/main/shell/chat.ts` | 为常规聊天与快捷动作共用回复呈现流程：直接显示回退、逐段延迟显示、收尾屏障、取消隔离。 |

## 错误与取消

翻译失败、sidecar 返回错误、SSE 正常结束却没有音频，均视为当前段没有可用音频：记录既有语音错误信息、放行该段文字并继续后续段。一个失败项不能卡住已预合成的后续项。

开始新消息、聊天取消、宠物会话销毁或宠物切换时，旧轮次的 AbortController 和语音队列会同时失效。延迟显示回调必须带本轮标识，失效轮次不能再向 UI 推送文字或音频。

## 测试策略

1. 扩展 `speakableText` 单测：裸 URL、URL 与自然语言混排、仅 URL、Markdown 图片/链接、路径、UUID、长哈希和代码内容的清洗与保留行为。
2. 扩展 `voiceProvider` 单测：明确区分产生音频、清洗为空、翻译失败、TTS 失败和无音频完成。
3. 扩展 `speechSequencer` 单测：首个 PCM 前只放行一次文字、后句先完成时文字仍按顺序、跳过/失败项不会卡住队列、停止后不泄漏旧轮回调。
4. 扩展 `chat` 集成单测：语音未就绪时保持原 LLM 流式显示；语音可用时按音频放行文字；URL-only 段显示但不调用语音；最终 memory 更新与完成事件不早于所有显示段；`stream` 与 `batch` 均采用相同同步规则。
5. 最后运行 `pnpm test`、`pnpm typecheck`、`pnpm build`；由于变更涉及主进程与渲染进程之间的音频时序，额外以 `pnpm preview` 做一次真实启动冒烟检查。
