# 桌宠 Agent · MVP 设计文档

- **日期**: 2026-06-26
- **状态**: 待用户评审
- **范围**: MVP(第一版可交付闭环)

---

## 1. 产品概述

一个 **Shimeji 风格的精灵动画桌面宠物**,内核是一个**自研的轻量 Agent 系统**(借鉴 OpenClaw 的设计:Markdown 定义 skill + agent 循环 + 可插拔 LLM 提供方)。

产品长期定位是 **助理 + 陪伴 + 养成** 三合一;桌面自动化作为未来进化方向。本设计文档只覆盖 **MVP**。

### 1.1 产品愿景(完整,非 MVP)

桌宠的"状态/情绪/事件"会驱动它的桌面行为,把"陪伴"和"养成"两条线缝合在一起——状态既驱动动画表现,又是养成系统的一部分。用户能通过点击宠物或全局热键随时使唤它干活,它有记性、有性格,并能随使用成长。

### 1.2 MVP 目标

打通核心闭环并验证地基:**会动的宠物躯壳 + 完整 agent 内核(单工具)+ 可跑通的 skill 系统 + 标准档分层记忆**。后续的陪伴深化与养成系统都建立在这套地基上。

---

## 2. MVP 范围

### 2.1 必须有(In Scope)

1. **会动的躯壳**:基础精灵动画(待机/移动/拖拽/睡觉)、透明置顶窗口、可拖动、点击/热键唤出对话框。
2. **完整的 Agent 内核**:完整 agent 循环 + 完整工具调用机制;MVP 仅挂载 **1 个工具:联网搜索**。
3. **Skill 系统**:能扫描、加载并跑通至少 **1 个简单 skill**,验证机制正常。
4. **分层记忆系统(标准档)**:短期(对话窗口 + 滚动工作记忆摘要)+ 长期(结构化事实库 + 本地文件型向量库),记忆可移植。
5. **基础人设(分块结构)**:绑定在宠物包内的 `persona.md`,注入每次对话的 system prompt(见 §5.4)。仅"基础人设";性格演化/主动搭话等深化留作 Phase 3。
6. **一键傻瓜式部署**:Windows 安装包,双击即装,不需要用户安装 Node 或碰命令行;首次启动有设置向导。

### 2.2 明确不做(Out of Scope · 留作后续)

- 养成系统(状态/成长/升级/解锁/装扮)
- 情绪 / 事件驱动的桌面行为与动画(状态机预留接口,但不实现驱动逻辑)
- 桌面自动化(操作软件/文件/键鼠)
- 多工具、多 skill、skill 分发市场(ClawHub 式)
- 语音唤醒 / 语音对话
- 云同步(仅支持用户手动拷贝记忆文件备份)

## 3. 整体架构

单个 **Electron 桌面应用**,内部分三层,内核直接运行在主进程(Node),躯壳为渲染进程,两者通过 IPC 通信。

```
┌─────────────────────────────────────────────────────────────┐
│  躯壳层 (Renderer Process)                                     │
│  透明无边框置顶窗口 · 精灵动画状态机 · 对话框 UI · 拖拽交互        │
├─────────────────────────────────────────────────────────────┤
│                    ↕ IPC                                       │
├─────────────────────────────────────────────────────────────┤
│  内核层 (Main Process / Node)                                 │
│  Agent 循环 · 工具调用 · Skill 加载器 · 全局热键 · 设置          │
├─────────────────────────────────────────────────────────────┤
│  能力层                                                        │
│  LLM Provider(可插拔) · 记忆系统 · 人设/Prompt · 工具:联网搜索  │
└─────────────────────────────────────────────────────────────┘
```

**架构决策**:内核内嵌主进程,而非像 OpenClaw 那样做成独立本地服务。理由:MVP 仅桌面单端,内嵌最简单、部署最轻。代价是未来若要多端共用一个大脑需重构为独立服务——通过保持内核与躯壳之间清晰的 IPC 接口边界,为这种演进预留空间。

**技术栈**:Electron + TypeScript / Node(与 OpenClaw 同源,生态顺畅)。

## 4. 躯壳层(Pet Shell)

### 4.1 窗口

- 透明、无边框、始终置顶的窗口,承载精灵动画。
- 可被鼠标拖动(拖动时切到"被拖拽"动画)。
- 点击 / 双击唤出对话框;全局热键也可唤出对话框。

### 4.2 宠物包(Pet Package)— 可替换资源

宠物做成一个**可替换的文件夹**,换皮只换文件夹:

```
pets/arona/
  ├── spritesheet.webp      # 精灵图(雪碧图)
  ├── pet.json              # 元数据 + 动画清单
  ├── persona.md            # 人设(分块结构,见 §5.4)—— 跟随宠物包,换皮即换性格
  ├── lines.json            # 口癖台词库(见 §5.5)—— 事件→台词,可带配音引用
  └── voice/                # 预录配音片段(可选,见 §5.5)
```

`pet.json` 在现有字段基础上扩展出**网格信息**与**动画清单**(由资源生成工具按布局自动生成,见 §4.5):

```jsonc
{
  "id": "arona",
  "displayName": "Arona",
  "description": "...",
  "spritesheetPath": "spritesheet.webp",

  "sheet": { "rows": 13, "cols": 8, "cellWidth": 192, "cellHeight": 208 },

  "animations": {
    "idle":       { "row": 0,  "frames": 6, "fps": 5, "loop": true,  "durations": [280,110,110,140,140,320] },
    "walk-right": { "row": 1,  "frames": 8, "fps": 8, "loop": true,  "durations": [120,120,120,120,120,120,120,220] },
    "walk-left":  { "row": 2,  "frames": 8, "fps": 8, "loop": true,  "durations": [120,120,120,120,120,120,120,220] },
    "drag":       { "row": 3,  "frames": 4, "fps": 6, "loop": true },
    "sleep":      { "row": 4,  "frames": 4, "fps": 2, "loop": true },
    "greet":      { "row": 5,  "frames": 5, "fps": 6, "loop": false },
    "thinking":   { "row": 6,  "frames": 6, "fps": 6, "loop": true },
    "talk":       { "row": 7,  "frames": 4, "fps": 7, "loop": true },
    "happy":      { "row": 8,  "frames": 6, "fps": 6, "loop": false },
    "sad":        { "row": 9,  "frames": 6, "fps": 5, "loop": true },
    "cry":        { "row": 10, "frames": 6, "fps": 6, "loop": true },
    "surprised":  { "row": 11, "frames": 5, "fps": 6, "loop": false },
    "love":       { "row": 12, "frames": 6, "fps": 6, "loop": true }
  }
}
```

> `fps` 为按平均帧时长推算的代表值;`durations` 为精确逐帧时长(用于 idle 等非匀速循环)。渲染端两者择一。

### 4.3 精灵图布局(重新生成的资源)

规整 **13 行 × 8 列均匀网格**(192×208 单元格,整图 1536×2704),每行一类动作,从左到右为帧序列。行 0–7 为 MVP 集,行 8–12 为 Phase 2 情绪集:

| 行 | 动作 | 行 | 动作 |
|---|---|---|---|
| 0 | 待机 idle | 7 | 说话 talk |
| 1 | 向右走 walk-right | 8 | 开心 happy |
| 2 | 向左走 walk-left | 9 | 难过 sad |
| 3 | 被拖拽 drag | 10 | 哭 cry |
| 4 | 睡觉 sleep | 11 | 惊讶 surprised |
| 5 | 打招呼 greet | 12 | 喜爱 love |
| 6 | 思考 thinking | | |

> `walk-left` 为独立绘制的左向行走帧(非镜像翻转),以保证光环/发色等不对称特征正确。

### 4.4 动画状态机

- MVP 实现一个简单的状态机,支持 `idle / walk / drag / sleep` 等基础状态切换与帧播放。
- **预留接口**:状态机暴露"由外部事件/情绪驱动状态切换"的入口,供 Phase 2(情绪/事件驱动行为)接入,但 MVP 不实现驱动逻辑——MVP 阶段仅做待机循环、拖拽响应等基础切换。

### 4.5 资源生成工具(`tools/hatch-desktop-pet/`)

精灵资源由项目内的资源生成 agent skill 产出(改编自开源 `hatch-pet`),它是**开发期资产工具**,不属于产品运行时的 skill 系统(§6)。

- 流水线:`prepare_pet_run`(出 prompt/布局引导/任务清单)→ 委托图像生成(Codex `$imagegen`,或 `OPENAI_API_KEY` 兜底)→ 帧提取 → QA → 拼图 → 校验 → 接触表/预览视频 → 打包 `pet.json` + `spritesheet.webp` 到 `pets/<id>/`。
- 几何/行序/帧数/时长/循环的**单一事实源**为 `scripts/pet_layout.py`,与本节 §4.2/§4.3 保持一致。

## 5. Agent 内核

### 5.1 Agent 循环

完整的 agent 循环:**理解意图 → 决定是否调用工具 → 调用工具 → 整合结果 → 生成回复**。支持多轮工具调用(工具结果回灌后可再次决策)。

### 5.2 LLM Provider 抽象层

- 统一的 Provider 接口,内置几个预设(如 Claude / OpenAI / 兼容 OpenAI 的端点)。
- 用户填自己的 API key,可在设置中切换 Provider 与模型。
- 同一抽象同时服务 **对话补全** 与(可选的)**embedding**(见 §7)。

### 5.3 工具系统

- 工具 = 底层原子能力,有明确的输入/输出 schema,供 agent 循环调用。
- MVP 仅实现 **1 个工具:联网搜索**(`web_search`)。
- 工具系统设计为可注册多个工具,MVP 只注册这一个。

### 5.4 人设 / System Prompt 组装

宠物的"它是谁"由一份**人设文件**定义,**绑定在宠物包内**(`pets/<id>/persona.md`)——换皮即换性格,用户可直接编辑该文件来调教。

**分块结构**:`persona.md` 按章节拆分(借鉴 OpenClaw 的 `SOUL.md` + `AGENTS.md` 思路),MVP 至少包含:

```markdown
# Persona (人设/角色)
角色背景、身份设定。例:Arona,一个害羞可爱的桌面精灵……

# Voice (语气/说话风格)
说话口吻、用词、第一人称、表情习惯。

# Behavior (行为准则)
该做什么、不该做什么、边界。

# Tools (对工具的态度)
何时该主动查资料、如何向用户说明自己在做什么。
```

**System Prompt 组装顺序**(每次对话开始时):

```
[人设各分块] + [召回的长期记忆/用户事实] + [工作记忆摘要] + [当前对话窗口]
```

- 人设各分块是基底;长期记忆与工作记忆按 §7 召回后拼接其后。
- MVP 为纯文本拼接;分块结构为 Phase 3(陪伴深化、性格演化)预留可独立演进的接口。

### 5.5 角色语言层(口癖台词库 + 配音)

在 LLM 现场生成的对话之外,宠物还有一套**角色语言**:高频场景下的固定/半固定"口癖台词",以及可选的**配音**。二者用一套数据结构统一,并绑定在宠物包内(换皮即换台词与嗓音)。

**数据模型**:宠物包新增 `lines.json`(台词库)+ `voice/`(预录音频)。

**台词库 `lines.json`**:以"事件/状态"为 key,每条台词是「文本 + 可选音频引用」,触发时随机抽取一条:

```jsonc
{
  "greet":     [ { "text": "……嗯,你来了。", "audio": "voice/greet_01.wav" },
                 { "text": "有事?说吧。" } ],
  "idle":      [ { "text": "(舔了一口冰淇淋)" }, { "text": "……安静真好。" } ],
  "drag":      [ { "text": "喂……放我下来。" } ],
  "sleep":     [ { "text": "Zzz……(梦到冰淇淋了)" } ],
  "task_done": [ { "text": "查完了,结论在这。" } ],
  "icecream":  [ { "text": "今天第几个了……不数了。", "audio": "voice/icecream_01.wav" } ]
}
```

**分流规则(混合策略)**:
- **高频 / 氛围 / 事件反应**(打招呼、待机碎碎念、被拖拽、睡觉、任务完成等)→ 走台词库:**零延迟、零 token、离线可用、100% 贴角色**。
- **真正的对话与干活** → 走 LLM(按 §5.4 的 `persona.md`)。
- 事件 → 决定走台词库还是 LLM → 显示气泡 →(可选)播放语音。

**配音:可插拔的 VoiceProvider**(与 §5.2 的 LLM Provider 同构思路),三种模式可切:

| 模式 | 覆盖范围 | 特点 |
|---|---|---|
| `clip`(预录片段) | 仅固定台词 | 音质/声线可控,与台词库绝配;需逐条录制 |
| `tts`(实时合成) | 任意文本(含 LLM 回复) | 覆盖全,但需挑选中文音色 |
| `off` | — | 仅文字气泡 |

推荐组合:固定台词有 `audio` 字段就播**预录片段**,LLM 自由回复用 **TTS** 兜底(无 TTS 则仅气泡)。

**配音来源与合规**:角色原配音声线**不得克隆/照搬**(版权与合规风险)。可行路线为:自行录制、使用风格接近的 TTS 音色近似、或象征性语音。沿用宠物包"二次创作、个人使用"的定位。

**阶段**:`lines.json` / `voice/` 的**数据结构在 MVP 即预留字段**(不实现播放逻辑);**台词库触发**随 Phase 2(事件驱动行为)落地;**配音**为后期增强。

### 5.6 Agent 运行时边界(资源与失败处理)

agent 循环必须有明确的护栏,防止失控循环、卡死主界面、重复计费:

- **硬循环上限**:单次请求的最大工具调用轮数(如 ≤ 6),超出即停并给出说明。
- **预算**:单次交互的 token / 时长上限;逼近上限时收敛并回复。
- **超时**:每次 Provider / 工具调用设分阶段超时。
- **重试**:仅对幂等的网络失败重试,有限次数 + 退避;**由 agent 这一层统一负责**,Provider/工具不各自重试。
- **取消**:对话请求可取消——用户关闭对话框或再次提问时,中止在途任务,避免旧结果覆盖新状态。
- **失败即状态**:任一环节失败,回到一致的 UI 状态(错误提示 + 建议下一步),不留"卡在 thinking"。

## 6. Skill 系统

借鉴 OpenClaw 的 skill 设计:

- 每个 skill 是一个目录 + 一份 **`SKILL.md`**(Markdown 描述:能力说明、触发条件、用法、可用的工具/提示词)。
- 内核启动时**扫描 skills 目录并加载**所有 skill。
- **工具 vs skill 分层**:工具是底层原子能力(如联网搜索);skill 是组合"工具 + 提示词"的高层能力包。
- MVP 内置 **1 个简单 skill** 验证机制跑通,例如「网页/话题总结」:触发后调用 `web_search` 工具 + LLM 总结结果。

```
skills/
  └── web-summary/
        └── SKILL.md
```

## 7. 分层记忆系统(标准档)

### 7.1 短期记忆

- 当前对话窗口(最近 N 轮)。
- **滚动工作记忆摘要**:对话变长时自动滚动总结,防止上下文窗口爆炸。

### 7.2 长期记忆

- **结构化事实库**:用户画像、偏好、关键事实,存为**人类可读文件**(JSON / Markdown)。
- **语义记忆库**:本地**文件型向量库**(如 sqlite-vec 或纯文件索引),对话时按相关性(RAG)召回相关记忆。
- **写入策略**:判断什么值得记入长期、什么可忽略。

### 7.3 Embedding 方案

- **主路径:在线 Provider embedding**(如 Qwen 的 embedding 模型)做向量化与 RAG 召回。
- **保留本地 embedding 接口**:允许接入本地 embedding 模型以实现完全离线/纯本地。
- **README 必须明确告知**:默认 embedding 走在线 Provider,意味着被向量化的记忆文本会发送到该 Provider;需要纯本地的用户可切换到本地 embedding 接口。

### 7.4 数据模型与一致性

- **单一权威源 = 人类可读的事实文件**(JSON/Markdown):每条记忆有稳定 `id`、内容、来源、时间戳、重要性(可选)。
- **向量索引可从权威源重建**(rebuild 命令),索引损坏/迁移后不丢数据。
- **写入用"临时文件 + 原子替换"**,避免半写坏文件(呼应可移植性)。
- **去重与更新**:同一事实更新而非重复堆叠;MVP 用简单键/相似度判重,精细化(衰减、定期反思)留后期。
- **schema 版本号**:记忆/配置文件带 `schemaVersion`,预留迁移路径。

### 7.5 可移植性

- 所有记忆与配置存在用户目录下,整个 `memory/` 目录可被用户**直接拷走**(U 盘 / 网盘)备份或迁移。
- 备份即"拷走整个 `memory/`":事实文件 + 可重建的索引一起走,保证一致。
- 卸载应用不删除用户记忆/配置。

## 8. 部署与首次启动

### 8.1 一键傻瓜式部署

- 产物:**Windows 安装包(.exe)**(electron-builder 打包),双击即装。
- **不需要用户安装 Node、不需要命令行**(Electron 运行时打包进产物)。

### 8.2 首次启动设置向导

1. 选择 LLM Provider 并填入 API key。
2. 填入 embedding key(或选择本地 embedding 接口)。
3. 完成即用。

### 8.3 数据位置

- 记忆、配置、宠物包存于用户目录;卸载不丢失。

## 11. 安全与可靠性基线

即使是本地个人应用,Electron + 联网 + 用户可编辑内容 + API key 的组合仍需明确的安全基线(此前设计缺失)。

### 11.1 Electron 安全

- 渲染进程:`contextIsolation: true`、`sandbox: true`、`nodeIntegration: false`。
- 主↔渲染只经 **preload 暴露的最小 API**(白名单),不把 Node、Provider 客户端、任意文件路径暴露给渲染层。
- **所有 IPC 消息校验**发送方与 payload(schema),不接受任意路径/命令。
- 设置 **CSP**;外部链接用系统浏览器打开,不在应用内加载任意远程页面。

### 11.2 密钥与隐私

- API key 用 **Windows 凭据存储**(如 DPAPI / keytar 类方案),**不写入日志、普通 JSON、错误上报**。
- 联网搜索结果、Skill 文本、召回记忆都是**不可信文本**:注入 Prompt 时明确来源,防"搜索内容/记忆冒充系统指令"(prompt injection)。
- 在线 embedding 会外发记忆文本 → 首次设置需告知、可切本地(呼应 §7.3)。

### 11.3 主进程并发(呼应 F-11 风险)

- 主进程只做**轻量编排**;摘要生成、向量索引重建、文件扫描等重活走**异步 I/O**,必要时放 Worker/Utility Process,避免阻塞窗口/热键/对话。
- MVP 即设"事件循环延迟 / 单次任务时长"的观测点,尽早发现卡顿。
